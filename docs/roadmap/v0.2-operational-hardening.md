# v0.2 Operational Hardening Roadmap

Status date: 2026-05-20

This roadmap records the current operational gaps discovered during the institutional trading engine review and turns them into trackable implementation work.

---

## Executive Summary

The current architecture is strong: scanner, bridge, orchestrator, governance, decision logs, signal lifecycle handling and adaptive expectancy are structurally in place.

The next work must focus on operational completeness instead of adding more intelligence features. The engine becomes materially more useful only when runtime reliability, real portfolio state, historical validation and documentation accuracy are tightened.

---

## Priority 1 — Stabilize `entry-exit-watcher`

### Problem

Recent workflow-error reports showed repeated failed `entry-exit-watcher` runs. This workflow is operationally critical because it monitors open signals, entry triggers, stops, targets, alerts and signal lifecycle updates.

If the watcher is unstable, the runtime layer becomes blind even when the decision engine is correct.

### Implementation Tasks

- Add explicit runtime input validation before executing the watcher.
- Fail fast when `POLYGON_API_KEY` is missing.
- Validate that the configured signals file exists.
- Add a unique watcher cycle id to logs.
- Make failure causes visible in GitHub Actions logs.
- Send a Telegram failure notification when Telegram secrets are configured.
- Preserve generated artifacts even on failure where possible.
- Keep the watcher deterministic and non-silent.

### Acceptance Criteria

- Manual workflow dispatch fails with a clear error if the signal file is missing.
- Scheduled workflow fails with a clear error if `POLYGON_API_KEY` is missing.
- Every run prints a `WATCHER_CYCLE_ID`.
- Telegram receives a failure message if the workflow fails and Telegram secrets are configured.
- Five consecutive watcher runs complete successfully after the fix in the real environment.

### Tracking

Status: In progress

First implementation target: `.github/workflows/entry-exit-watcher.yml`

## Priority 2 — Add Portfolio State for Real Governance

### Problem

Governance includes drawdown and risk-limit concepts, but live runtime currently cannot enforce real portfolio drawdown limits without persisted portfolio state.

A hard-coded `portfolio_drawdown_percent = 0.0` means the drawdown kill switch is structurally present but not operationally active.

### Implementation Tasks

- Add `data/portfolio_state.json` schema.
- Add `PortfolioState` dataclass/model.
- Add `PortfolioStateStore` reader/writer.
- Connect portfolio state to `live_runtime_cycle.py`.
- Add conservative fallback when state is missing.
- Add tests for drawdown and daily-loss kill switch behavior.

### Minimal State Shape

```json
{
  "equity_start": 10000.0,
  "equity_current": 9720.0,
  "drawdown_percent": -2.8,
  "daily_loss_percent": -0.9,
  "open_positions": [],
  "updated_at": "2026-05-20T22:00:00+02:00"
}
```

### Acceptance Criteria

- Governance receives real portfolio values.
- Drawdown kill switch no longer depends on a hard-coded zero value.
- Missing state produces a visible conservative warning.
- Tests cover normal, missing and corrupted portfolio state.

Status: Planned

## Priority 3 — Sync Roadmap and Documentation with Code Reality

### Problem

Some roadmap statements describe already implemented features as future work. This creates architecture drift and makes onboarding misleading.

### Implementation Tasks

- Review README roadmap against current implementation.
- Split roadmap into `Done`, `In progress`, `Planned`, and `Known limitations`.
- Document that portfolio governance is structurally implemented but still needs live portfolio-state integration.
- Document watcher stability as operational prerequisite.

### Acceptance Criteria

- README does not list implemented features as future work.
- Current limitations are explicit and not hidden.
- New developers can identify what is production-ready and what is not.

Status: Planned

## Priority 4 — Connect Backtesting to Real Historical Data

### Problem

Backtesting logic exists, but without real historical data validation it mainly proves code behavior, not market usefulness.

### Implementation Tasks

- Add Polygon historical aggregate fetcher using `/v2/aggs/ticker/...`.
- Persist historical bars under a deterministic cache path.
- Run setup-score snapshots against future 5/10/20-day returns.
- Produce `reports/backtests/backtest_summary.json`.
- Add summary metrics: sample size, win rate, average return, max adverse excursion and false-positive rate.

### Acceptance Criteria

- Backtest can run for at least the default symbol universe.
- Results are based on real historical Polygon bars, not mocks.
- Output is machine-readable and committed or uploaded as an artifact.

Status: Planned

---

## Priority 5 — Expand Symbol Universe Beyond Tech

### Problem

The current universe is too narrow and tech-heavy for an institutional market-regime scanner.

### Minimum Expansion

- Core indices: `SPY`, `QQQ`, `IWM`, `DIA`
- Rates / bonds: `TLT`, `IEF`, `SHY`
- Sectors: `XLK`, `XLF`, `XLE`, `XLV`, `XLY`, `XLP`, `XLI`
- Mega caps: `AAPL`, `MSFT`, `NVDA`, `AMZN`, `GOOGL`, `META`, `TSLA`
- Semiconductors: `SMH`, `MU`, `AMD`, `AVGO`
- Commodities: `GLD`, `SLV`, `USO`

### Acceptance Criteria

- Universe config is centralized.
- Sector breadth and risk-on/risk-off interpretation improve.
- Tests cover the expanded universe configuration.

Status: Planned

---

## Priority 6 — Automate Expectancy Feedback Reports

### Problem

Adaptive expectancy exists but needs a stronger automated reporting loop.

### Implementation Tasks

- Add weekly GitHub Action for expectancy report generation.
- Feed outcome tracking into expectancy updates automatically.
- Send weekly Telegram expectancy summary.
- Persist expectancy report artifacts.

### Acceptance Criteria

- Weekly expectancy report runs without manual intervention.
- Telegram summary includes evaluated samples, strongest setups, weakest setups and data-quality warnings.
- Expectancy reports are auditable over time.

Status: Planned

## Recommended v0.2 Sequence

1. `fix(actions): stabilize entry-exit watcher runtime validation`
2. `feat(runtime): add portfolio state store for governance kill switch`
3. `docs(roadmap): sync roadmap with implemented code reality`
4. `feat(backtest): validate setup scores with polygon historical bars`
5. `feat(scanner): expand symbol universe for sector and cross-asset coverage`
6. `ci(expectancy): add weekly adaptive expectancy report workflow`

---

## Non-Goals for v0.2

- Broker execution
- Fully automated live trading
- Real-time websocket engine
- Machine-learning inference layer
- Dashboard UI

These can come later. v0.2 must first make the current decision-support system operationally reliable and measurable.
