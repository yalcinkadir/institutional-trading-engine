# ARCH106 Strict Module Classification Policy

ARCH106 does not treat every source module as required production logic. The goal is to make the runtime path smaller, clearer and more auditable by separating proven production modules from support, research and removal candidates.

## Classification meanings

| Classification | Meaning | Required evidence |
| --- | --- | --- |
| `connected_runtime` | Module is part of a production-relevant runtime path. | Static import/reachability plus CI-backed runtime execution proof. |
| `experimental` | Module may be useful for research, reporting support or future product layers, but is not runtime-proven. | Clear ownership/purpose and no claim of production execution. Runtime fields may remain `null`. |
| `test_only` | Module is only used by tests or fixtures. | Import/use evidence from tests and no production runtime claim. |
| `quarantine` | Module is risky, stale, duplicated or unclear and must not be used without review. | Rationale explaining why the module is isolated. |
| `delete_candidate` | Module appears unnecessary and can be removed after a deletion PR proves no breakage. | No production/test references or an explicit replacement path. Deletion still requires CI proof. |

## Hard rules

1. Do not classify a module as `connected_runtime` without a runtime execution proof test.
2. Do not delete a module in the same step where it is first identified as a candidate.
3. Prefer `experimental` over `connected_runtime` when a module is useful but not executed by a proven runtime path.
4. Prefer `quarantine` over `delete_candidate` when risk is unclear or the module may contain unique logic.
5. A `delete_candidate` classification is not deletion approval. It only opens the path for a separate removal PR.
6. Inventory counters must be regenerated by `scripts/generate_module_inventory.py`, not edited manually.
7. Repo-writing promotion workflows must use repo-wide concurrency and upload evidence artifacts with `if: always()`.

## Practical triage order

1. Runtime-critical modules with proof -> `connected_runtime`.
2. Useful but unproven modules -> `experimental`.
3. Test-only helpers -> `test_only`.
4. Risky, duplicated or unclear modules -> `quarantine`.
5. Unreferenced and replaceable modules -> `delete_candidate`.

## ARCH106 issue gate

Issue #106 remains open until the legacy set is reduced into explicit classifications and all production-critical paths have runtime execution evidence. Reducing the legacy count is not enough; the classification must be truthful.
