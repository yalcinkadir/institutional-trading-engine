# IP2 Public Repository Hygiene and Private Edge Handling Policy

This policy defines how the public repository should be maintained without exposing proprietary strategy edge, confidential evidence artifacts or production-like execution parameters.

## Policy status

```text
Policy ID: IP2
Scope: public repository hygiene and private edge handling
Mode: mandatory PR review guidance
Trading authorization: none
```

This document is operational guidance. It does not grant live trading permission and does not replace legal review.

## Public repository may contain

The public repository may contain framework and research infrastructure:

- architecture and orchestration code
- deterministic validation engines
- paper-only broker and execution interfaces
- demo defaults and synthetic examples
- tests and fixtures
- public operations documentation
- CI and quality gates
- audit/report-generation machinery
- public-safe runbooks

Public code must remain useful as a framework even when private modules and private configuration are absent.

## Public repository must not contain

The public repository must not contain private edge or sensitive operational material:

- real decision thresholds
- real setup rankings or proprietary setup maps
- production scoring weights
- confidential entry/exit profiles
- private model factors
- real ranked opportunity reports
- raw private evidence artifacts
- provider credentials, API keys, tokens or broker secrets
- production account identifiers
- manual notes that reveal the actual edge formula

## Demo defaults rule

When a public module needs strategy-like configuration, use one of these patterns:

1. clearly marked demo defaults
2. synthetic fixtures
3. external configuration interfaces
4. optional private imports that fail safely when unavailable

Public defaults must not be tuned to represent the real production edge.

## PR review checklist

Every PR that touches strategy, scoring, thresholds, execution, reports or evidence must answer:

```text
Does this add a real threshold?
Does this reveal scoring weights?
Does this expose setup ranking logic?
Does this include real opportunity/ranking reports?
Does this include raw private evidence?
Does this include provider, broker or account information?
Does this bypass the IP boundary scanner?
Does this weaken paper-only/no-live-trading language?
```

If any answer is yes, the PR should be blocked until the content is removed, made synthetic, or moved behind a private boundary.

## Telegram and report policy

Telegram/report delivery is allowed only as research and paper-observation reporting.

Allowed:

- daily evidence summaries
- backtest summaries
- paper observation summaries
- execution quality summaries
- kill-switch state summaries
- high-level risk and drift warnings

Not allowed:

- live buy/sell instructions
- order buttons
- broker execution links
- private threshold values
- real scoring weights
- ranked opportunity lists containing private edge
- text implying live trading authorization

Every Telegram/report message must be compatible with this footer:

```text
Research / Paper Observation Only. No live trading authorization.
```

## Private edge handling

Private edge should be handled outside public source control:

- local private modules
- private repository
- private configuration store
- environment variables for non-secret config
- secret manager for credentials

The public code should access private edge only through explicit interfaces. Missing private edge should result in a safe demo/default path or a fail-closed validation error.

## Artifact hygiene

Generated reports and raw evidence should not be committed by default.

Allowed in public:

- synthetic examples
- small demo fixtures
- schema examples
- redacted reports

Private or generated by default:

- real ranked opportunities
- real observation logs
- real raw fills/orders
- real provider extracts
- real evidence baselines
- local databases
- report archives

## Required checks

Before merge:

```bash
python scripts/check_ip_boundary.py --root . --no-write
pytest tests/test_ip_boundary.py -q
```

When available, the full regression suite should remain green:

```bash
pytest -q
```

## Escalation rule

If a private-edge leak is found after merge:

1. stop further work on top of the leaked commit
2. remove or redact the leaked content
3. rotate any affected secret if credentials were involved
4. add a regression test or policy term that prevents recurrence
5. document the correction in the PR or incident notes

## Non-goals

This policy does not:

- enable live trading
- certify legal IP protection
- certify financial performance
- replace risk review
- replace security review

It is a practical engineering guardrail for keeping the public repository safe while continuing institutional framework development.
